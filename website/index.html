Auditable compute over Vortex-native data.

Why ShardLoom exists

Make execution visible instead of magical.

Many local engines are excellent at direct-file execution. ShardLoom is targeting the larger workflow around a run: admission, preparation, native Vortex paths, result-sink proof, no-fallback policy, and claim boundaries that a human or agent can inspect.

What this is

  • a Vortex-native local compute foundation
  • a no-fallback evidence surface
  • a pre-optimization runtime-development lane
  • a public status board for supported, blocked, and report-only paths

What this is not

  • not a public speed leaderboard
  • not an Apache Spark substitute claim
  • not a production SQL/DataFrame platform claim
  • not a production object-store, lakehouse, or Foundry claim

Current state

Local compute foundation, not a finished platform claim.

ShardLoom's public posture is intentionally narrow: explicit execution-mode reporting, prepared/native Vortex evidence, no-fallback enforcement, and deterministic unsupported diagnostics.

        Runtime lanes

        Separate how data enters from what workload semantics mean.

        Execution modes describe source/preparation lanes. Engine modes describe batch, live, or hybrid workload semantics. The website keeps these separate so benchmark readers do not infer a hidden fast mode or production live/hybrid runtime.

        Execution

        compatibility_import_certified

        Use for ingest/stage certification. Timing includes compatibility parse, import, Vortex write/reopen, scan, result, and evidence work.

        Execution

        prepared_vortex

        Use for current prepared-artifact benchmark evidence. It is the preferred performance-development lane, not a public performance claim.

        Execution

        native_vortex

        Use for scoped existing Vortex artifacts with visible admission, source-backed scan, Native I/O, and no-fallback fields.

        Engine

        batch / live / hybrid

        Batch is the local foundation. Live and hybrid have fixture evidence and contracts, but no broker, durable state, or production platform claim.

        Claim boundaries

        What the website does not claim.

        These boundaries keep the public site aligned with the repository evidence.

        No Apache Spark substitute claimExternal engines appear only as baselines or references, never fallback execution.
        No production SQL/DataFrame claimLanguage and adapter surfaces remain gated unless implementation and evidence support them.
        No production object-store/lakehouse/Foundry claimThose surfaces remain blocked, report-only, or planned until future proof exists.
        No performance or superiority claimSmoke measurements and local comparisons are attribution evidence, not public ranking proof.
