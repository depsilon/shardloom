Field Guide

A compact Starlight docs shell for ShardLoom’s current public surface. Start with local proof, Python route shape, benchmark methodology, and limitations, then use the vocabulary atlas for exact route and evidence terms.

Start Here

Section titled “Start Here”
  • What is ShardLoom? - A pre-release compute engine focused on Vortex-prepared routes, explicit evidence, and no hidden fallback.
  • Evidence-gated compute - Every supported route must emit evidence before any claim can be upgraded.
  • No fallback - Unsupported work blocks or reports a deterministic diagnostic instead of running through another engine.

Execution Routes

Section titled “Execution Routes”
  • compatibility_import_certified - The certified cold ingest/stage route for local compatibility inputs and evidence-heavy runs, including source-state layout and digest attribution.
  • direct_compatibility_transient - A scoped one-shot local route without persistent Vortex preparation.
  • Generated source route - No input dataset is read; ShardLoom creates deterministic rows and writes local outputs with evidence.

UniversalIngress

Section titled “UniversalIngress”
  • UniversalIngress - The source admission layer that recognizes, admits, or blocks every potential input family.
  • SourceState - Reusable source identity, schema, fingerprint, adapter posture, and source evidence.
  • Source adapter status - The visible admit/block posture for an input adapter and source format.
  • Local SQLite fixture - A named local SQLite table scan to workspace-safe JSONL plus roundtrip SQLite replay, without SQL pushdown or network effects.

Evidence + Certificates

Section titled “Evidence + Certificates”
  • Effectful operation admission - The matrix that admits local fixture/metadata exceptions while blocking network, credential, plugin, UDF, API, model, and vector effects by default.
  • claim_gate_status - The field that says whether evidence is claim-grade, fixture-smoke-only, report-only, or blocked.
  • Native I/O certificate - Evidence for admitted local source or sink I/O without external engine fallback.
  • Materialization boundary - Evidence describing when data was decoded, materialized, or kept source-backed.
  • Result-sink replay - Output proof that a local result sink can be replayed or checked for evidence consistency.

Vortex Ingest

Section titled “Vortex Ingest”
  • vortex_ingest - The prepare-once stage that turns admitted non-Vortex sources into VortexPreparedState.
  • VortexPreparedState - The prepared artifact/state that prepared_vortex executes from.
  • Prepared state reuse - Reuse of a prepared Vortex artifact across scenarios, queries, or outputs.

Prepared/Native Vortex

Section titled “Prepared/Native Vortex”
  • prepared_vortex - Prepared warm execution over an existing VortexPreparedState, not direct non-Vortex input.
  • native_vortex - Execution over input that already exists as Vortex before the query starts.
  • Source-backed scan - Prepared/native scan evidence showing columns, pushdown, decode, and materialization posture.

Benchmarks

Section titled “Benchmarks”
  • Benchmark evidence - Committed local timing and coverage artifacts interpreted as evidence, not a leaderboard.
  • Certified cold route - Timing scope that includes source read, parse, source-state materialization, Vortex array build/write/digest/reopen, compute, output, and evidence.
  • Prepared warm route - Timing scope for query/runtime after VortexPreparedState already exists.
  • external_baseline_only - Competitor engines are comparison context only and never satisfy ShardLoom evidence gates.

I/O + Outputs

Section titled “I/O + Outputs”
  • OutputPlan - The output planning layer that stays separate from input format and execution route.
  • SinkArtifact - The written output artifact plus digest, format, replay, and certificate posture.
  • Output fanout - One prepared source or result writing multiple output formats without coupling input and output types.

Scale + Resource Envelope

Section titled “Scale + Resource Envelope”
  • Scale classes - Bounded scale claims such as local smoke, local claim-grade, split-local, object-store runtime, and distributed runtime.

Platform Boundaries

Section titled “Platform Boundaries”
  • Object-store boundary - S3/GCS/ADLS now have a public no-credential fixture read smoke, while live provider runtime remains separately blocked.
  • Table/lakehouse boundary - ShardLoom can read local Iceberg metadata, manifest summaries, manifest split plans, and explicitly requested local Parquet Iceberg data-file scans; broad lakehouse runtime remains gated.
  • Foundry boundary - Local/dev-stack Foundry proof posture without production Foundry runtime or Spark substitution claims.

Unsupported Diagnostics

Section titled “Unsupported Diagnostics”
  • Deterministic blockers - Unsupported work must return an explicit blocker instead of silently disappearing or falling back.
  • report_only - A visible design or capability posture that is not runtime execution.

Claim Boundary

Section titled “Claim Boundary”

The Field Guide explains vocabulary. It does not create a runtime, performance, production, SQL/DataFrame, object-store, lakehouse, Foundry, package-publication, Spark-displacement, or fallback-execution claim.
