No Fallback

ShardLoom must not silently delegate unsupported execution to another query engine.

In this dossier

Why it matters How ShardLoom uses it What it proves Claim boundary Source docs

Why it matters

The policy keeps evidence auditable: unsupported work becomes a deterministic diagnostic instead of an invisible execution handoff.

How ShardLoom uses it

CLI, benchmark, and evidence rows expose fallback and external-engine fields so readers can see whether ShardLoom executed the supported path.

What it proves

A supported row can show `fallback_attempted=false` and `external_engine_invoked=false` for its scoped workload.

What it does not prove

It does not prove every SQL, DataFrame, object-store, lakehouse, or adapter path is implemented.

Evidence fields

  • fallback_attempted
  • external_engine_invoked
  • claim_gate_status
  • diagnostic_code

Claim boundary

No-fallback evidence is workload-scoped and does not create a broad production platform claim.
