Plain-English meaning

The object-store boundary keeps S3, GCS, and ADLS parsing/planning separate from runtime reads, writes, credentials, and commits.

Why it matters

This concept helps users interpret ShardLoom's i/o and output posture without reading the phase plan or RFCs first.

How ShardLoom uses it

ShardLoom surfaces this through docs, capability/status pages, use cases, or evidence fields such as credential_policy_status, object_store_io, network_probe_allowed, fallback_attempted.

Current support

Current status is blocked-report-only. Treat that status exactly as shown; scoped, smoke, report-only, planned, blocked, or unsupported posture must not be read as broader runtime support.

Evidence fields

It helps identify the exact evidence or documentation surface that applies to this concept.

  • credential_policy_status
  • object_store_io
  • network_probe_allowed
  • fallback_attempted

Claim boundary

What it does not claim

Object-store URI recognition or planning does not imply object-store runtime support.
