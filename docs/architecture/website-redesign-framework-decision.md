# Website Redesign Framework Decision

Status: accepted and implemented for `GAR-WEB-REDESIGN-2I`.

Decision status: `approved_runtime_static_site_migration`.

Recommendation: use Astro for the public website shell and Starlight for documentation/Field Guide
surfaces, while preserving Cloudflare static deployment from `./website`.

Migration status: `implemented_after_explicit_approval`.

This decision does not add ShardLoom compute runtime behavior, website server runtime services,
benchmark recomputation, package publication, performance claims, production claims, or fallback
execution.

## Current Site Evidence

The public website now uses a controlled Astro/Starlight static build:

- `website-src/package.json` owns the website-only Node toolchain and lockfile.
- `website-src/astro.config.mjs` uses static output and writes the built site to `../website`.
- `website-src/src/content.config.ts` defines schema-backed content collections for docs, use cases,
  and status rows.
- `website-src/src/data/` contains the committed website data snapshots for Field Guide, use cases,
  status rows, benchmark evidence, and benchmark manifest.
- `website-src/scripts/sync-content.mjs` generates Starlight docs content and collection entries
  from the committed data snapshots before build.
- `website-src/scripts/postbuild-static.mjs` writes legacy `.html` compatibility pages after Astro
  emits directory routes.
- `website-public/` contains static assets copied into the built site, including the ShardLoom logo
  assets, benchmark artifacts, `_headers`, `_redirects`, and `validate_static_assets.js`.
- `wrangler.toml` continues to serve `./website` as Cloudflare Workers Static Assets.
- `website/pagefind/` is generated as a local static-search bundle by Starlight; it is website-only
  and does not call an external search service.

## Decision

Use Astro + Starlight going forward.

Rationale:

- ShardLoom's public site now needs a durable content model for Field Guide terms, use cases,
  status rows, route definitions, benchmark interpretation, and source-linked references.
- Starlight provides documentation navigation, local search, accessible docs layout, and static
  search primitives without a server runtime.
- Astro custom pages preserve the product-grade homepage, benchmark dashboard, architecture page,
  status matrix, and use-case browser.
- The built artifact remains inspectable under `website/`, and the deployment path remains the
  existing Cloudflare static asset directory.

## Implementation Shape

- Astro custom pages own `/`, `/start`, `/use-cases`, `/use-cases/<id>`, `/benchmarks`,
  `/architecture`, `/compute-engine-flow`, and `/status`.
- Starlight owns `/field-guide`, `/field-guide/<slug>`, `/docs`, and `/404`.
- Static benchmark pages render committed artifacts only.
- Legacy `.html` paths remain via post-build compatibility files and `_redirects`.
- Search is local Pagefind generated by Starlight; no external search SaaS is used.

## Boundaries

The migration must not:

- run benchmarks during site build;
- fetch live GitHub/raw content at runtime;
- add a Cloudflare Worker server runtime;
- add Astro/Starlight/Pagefind as ShardLoom compute runtime dependencies;
- hide blocked, report-only, unsupported, or not-planned capabilities;
- make performance, superiority, Spark-replacement, production, broad SQL/DataFrame,
  object-store/lakehouse, Foundry production, or package-publication claims;
- use external engines as fallback execution.

## Rollback Path

If the Astro/Starlight build becomes blocked, rollback is:

1. restore the last known good `website/` artifact from Git history;
2. keep `wrangler.toml` serving `./website`;
3. remove or pause the `website-src/` build path in a dedicated rollback PR;
4. preserve claim-safety validators before publishing.

## Claim Boundary

This is a website implementation decision only. It does not imply runtime support, benchmark
superiority, package publication, production readiness, Spark replacement, broad SQL/DataFrame
runtime, object-store/lakehouse runtime, Foundry production support, or external fallback execution.
