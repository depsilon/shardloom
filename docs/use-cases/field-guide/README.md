# Use Case Field Guide

This glossary explains terms that appear in the Use Case Atlas. Each term is educational and does
not upgrade support status.

## Terms

| Term | One-Sentence Explanation | Why It Matters | How To Inspect It | Related Use Cases | Reference Files |
| --- | --- | --- | --- | --- | --- |
| execution mode | The source/preparation route a ShardLoom run uses, such as compatibility import, prepared Vortex, native Vortex, direct transient, or auto. | It prevents compatibility ingest, prepared artifact runtime, and native runtime rows from being compared as the same thing. | Check `execution_mode` in CLI, Python, or benchmark evidence. | `compatibility-import-certified-local`, `prepared-native-vortex-runtime-direction` | `docs/architecture/compute-engine-flow-reference.md` |
| engine mode | The workload semantics layer: batch, live fixture, hybrid fixture, or report-only streaming posture. | It separates batch local evidence from live/hybrid claims. | Check engine selection/capability outputs and compute-flow docs. | `evidence-audit-claim-gates` | `docs/architecture/compute-engine-flow-reference.md`, `python/README.md` |
| Vortex-native | A ShardLoom path that treats Vortex as the native data representation or provider boundary. | It is core to the no-fallback architecture. | Inspect Native I/O and Vortex artifact fields. | `prepared-native-vortex-runtime-direction` | `docs/architecture/compute-engine-flow-reference.md` |
| compatibility import | A certification lane that reads compatibility input and imports it into Vortex-backed execution/evidence. | It carries workflow and evidence costs, so it is not pure query speed. | Inspect `compatibility_parse_millis` and `compatibility_to_vortex_import_millis`. | `compatibility-import-certified-local` | `docs/getting-started/certified-local-workload.md` |
| prepared Vortex | A runtime-development lane using prepared Vortex artifacts. | It is the main optimization direction. | Inspect `prepared_vortex` rows and source-backed scan fields. | `prepared-native-vortex-runtime-direction` | `benchmarks/traditional_analytics/README.md` |
| native Vortex | A lane that starts from Vortex input when admitted by the scenario. | It distinguishes native Vortex input from compatibility import. | Inspect `execution_mode=native_vortex` rows where present. | `prepared-native-vortex-runtime-direction` | `docs/architecture/compute-engine-flow-reference.md` |
| direct transient | A compatibility path that avoids Vortex persistence when admitted, but is not Vortex-native. | It prevents transient compatibility work from being mislabeled as native. | Check capability and compute-flow diagnostics. | `sql-dataframe-capability-posture` | `docs/architecture/compute-engine-flow-reference.md` |
| no fallback | Unsupported work must not silently run through another engine. | It protects evidence and user trust. | Look for `fallback_attempted=false` and `external_engine_invoked=false`. | `first-10-minutes-local-smoke`, `evidence-audit-claim-gates` | `README.md` |
| materialization boundary | The point where data is decoded or materialized instead of staying native/encoded. | It explains where work was avoided or paid. | Inspect `data_decoded`, `data_materialized`, and materialization fields. | `benchmark-interpretation-evidence-not-leaderboard` | `docs/architecture/compute-engine-flow-reference.md` |
| Native I/O certificate | Evidence that source or sink I/O stayed within the admitted native contract. | It gates native I/O claims. | Inspect `native_io_certificate_status` and output Native I/O fields. | `compatibility-import-certified-local`, `output-result-sink-and-fanout-boundary` | `docs/getting-started/certified-local-workload.md` |
| result-sink replay | A proof that the output sink can be replayed or verified. | It makes output evidence auditable. | Inspect `result_replay_verified` and result-sink fields. | `output-result-sink-and-fanout-boundary` | `docs/benchmarks/local-taxonomy-benchmark.md` |
| claim gate | The field that says which public claim, if any, the evidence supports. | It prevents benchmark or smoke rows from becoming unsupported claims. | Inspect `claim_gate_status`. | `evidence-audit-claim-gates` | `docs/architecture/operational-evidence-policy-hardening.md` |
| fixture smoke | A scoped, deterministic local proof that is useful but narrow. | It gives signal without claiming broad runtime support. | Check status and scenario/profile fields. | `messy-data-local-fixtures` | `docs/architecture/benchmark-suite-catalog.md` |
| report-only | A posture where users can inspect diagnostics or plans, but runtime execution is not supported. | It lets ShardLoom be explicit without overclaiming. | Inspect `support_status=report_only`. | `sql-dataframe-capability-posture` | `python/README.md` |
| external baseline | An outside engine used for comparison context only. | It must never become fallback execution. | Inspect benchmark docs and `external_engine_invoked=false`. | `benchmark-interpretation-evidence-not-leaderboard` | `docs/benchmarks/baseline-comparison-boundary.md` |
| residual-native | ShardLoom-native residual work after an admitted provider path. | It keeps residual compute visible instead of overclaiming provider support. | Inspect operator execution class and materialization fields. | `prepared-native-vortex-runtime-direction` | `benchmarks/traditional_analytics/README.md` |
| encoded-native | End-to-end encoded execution that avoids decode/materialization for admitted operators. | It is a future claim gate, not implied by scoped encoded-predicate evidence. | Inspect `encoded_native_claim_allowed`. | `prepared-native-vortex-runtime-direction` | `docs/architecture/compressed-encoded-kernel-registry.md` |
| source-state reuse | Reuse of source discovery, schema, parse, or source-state work. | It is a major runtime usability/performance direction. | Inspect `source_state_reuse_hit` and source-state fields. | `prepared-native-vortex-runtime-direction`, `output-result-sink-and-fanout-boundary` | `docs/architecture/io-reuse-and-fanout-architecture.md` |
| output-plan reuse | Reuse of sink/write planning independent of input format. | It enables future fanout without coupling input and output formats. | Inspect `output_plan_reuse_hit` when implemented. | `output-result-sink-and-fanout-boundary` | `docs/architecture/io-reuse-and-fanout-architecture.md` |
